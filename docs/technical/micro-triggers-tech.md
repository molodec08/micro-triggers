# Бриф: "Микро-триггеры внимания" — техническая часть

См. также: [бизнес-часть](../business/micro-triggers-brief.md).

Разработчик: React/TS/Vue на фронте, Node.js + PostgreSQL (Sequelize) на бэке.
Иностранная карта есть — платёжных ограничений нет.

## Технический стек
- Шаблон: **React Router app** (актуальная замена Remix-шаблона в Shopify CLI),
  НЕ extension-only — нужен бэкенд для настроек и Billing API
- Admin UI: Polaris — простые тумблеры вкл/выкл на каждый триггер, live-превью
- Storefront: Theme App Extension (Liquid + JS), вся логика триггеров — чистый
  клиентский JS, максимально лёгкий бандл
- БД: Postgres (в шаблоне по умолчанию SQLite — заменить провайдера в Prisma
  schema.prisma). Схема: shop → какие триггеры включены, тексты, задержки, файл звука
- Учитывать при тестировании: блокировщики рекламы/приватности в браузере иногда
  режут audio/visibility API

## Реализация триггеров (MVP v1)
1. **Мигающая вкладка браузера** при уходе с вкладки, если в корзине есть товар
   (`visibilitychange` → смена `document.title` и favicon, возврат при фокусе)
2. **Минималистичный exit-intent попап** (отслеживание движения мыши к верхней
   границе окна)
3. **Звуковой сигнал** при добавлении в корзину / на checkout (Web Audio API)

## Исправлено, 2026-07-20: тип `s-app-nav`

`@shopify/polaris-types@1.0.7` (актуальная опубликованная версия — апгрейд не
помогает) не объявляет `<s-app-nav>`, хотя это реальный, поддерживаемый рантаймом
компонент ("App nav web component", см. shopify.dev/docs/apps/design/navigation).
Остальные `s-*` теги регистрируются пакетом через
`declare module "react" { namespace JSX { interface IntrinsicElements {...} } }`
— `s-app-nav` там просто пропущен.

Временный патч в `env.d.ts`: локальный augmentation того же модуля `react` для
`s-app-nav`. Ключевой нюанс — `env.d.ts` должен быть модулем (иметь `import`),
иначе `declare module "react"` не мержится с существующими типами `@types/react`
и вместо этого либо не резолвится, либо ломает остальные экспорты `react`
(`useState` и т.д.). Добавлен `import type { DetailedHTMLProps, HTMLAttributes }
from "react"` в начало файла — после этого `npm run typecheck` проходит чисто.

Удалить этот патч, когда в `@shopify/polaris-types` появится нативное
объявление `s-app-nav`.

## Шаг 3 (Storefront: реализация трёх триггеров) — выполнено, 2026-07-20

**Theme App Extension** — `extensions/micro-triggers-storefront/`, сгенерирован
через `shopify app generate extension --template=theme_app_extension`. Демо-файлы
шаблона (`star_rating.liquid`, `stars.liquid`, `thumbs-up.png`) удалены.

Один **app embed block** (`blocks/app_embed.liquid`, `target = "body"`) —
мерчант включает его один раз в редакторе темы; блок подключает единственный
JS-файл `assets/micro-triggers.js` (без сборки, без зависимостей — весь код
трёх триггеров в одном файле на чистом ES5, чтобы бандл оставался лёгким).

**Передача настроек на сторфронт: App Proxy**, а не metafields — единственный
источник правды для настроек остаётся Postgres (см. шаг 2), без риска
рассинхронизации с дублирующим хранилищем.
- `shopify.app.toml`: добавлен `[app_proxy]` (`prefix = "apps"`,
  `subpath = "micro-triggers"`) и обязательный для него access scope
  `write_app_proxy` (ранее было `scopes = ""` — пересмотрено на шаге 3, так как
  App Proxy — не Admin GraphQL API, но тоже требует явного scope).
- `app/routes/proxy.settings.tsx` — публичный (без Admin-сессии) route,
  `authenticate.public.appProxy(request)` определяет `shop` по запросу,
  дальше читает три Prisma-модели и отдаёт единый JSON.

**Логика триггеров в `micro-triggers.js`**
1. **Мигающая вкладка** — `visibilitychange`; при уходе со страницы с
   непустой корзиной (`GET /cart.js`) чередует `document.title` с текстом
   настройки раз в секунду; возвращает исходный заголовок при фокусе.
2. **Exit-intent попап** — слушает `mouseleave` на `document`, срабатывает
   только когда курсор уходит через верхнюю границу (`clientY <= 20`) и в
   корзине есть товар; рисует минимальный оверлей (текст + промокод +
   кнопка закрытия) без сторонних библиотек.
3. **Звук** — Web Audio API (`AudioContext`, короткий синус-бип), без
   аудиофайлов. Слушает `submit` форм с `action` на `/cart/add`
   (добавление в корзину) и клики по ссылкам на `/checkout`, каждый — под
   своим флагом (`playOnAddCart`/`playOnCheckout`) из настроек.

**Учтено при написании**: весь фетч настроек и `/cart.js` обёрнут в
`try/catch`/`.catch()` — если блокировщик приватности режет `fetch`/`AudioContext`,
триггеры просто не активируются, без ошибок в консоли витрины.

**Не проверено вручную** (нужно сделать в шаге 5, QA): реальная работа на
dev store — прогонялась только статическая проверка (`npm run lint`,
`npm run typecheck`), реального рендеринга в браузере на теме Dawn/Horizon
ещё не было.

## Диагностика, 2026-07-20: редирект на список приложений вместо /app

**Симптом**: переход на `.../apps/simple-app-121/app` в Shopify admin сразу
редиректил на `.../settings/apps?tab=installed` (список установленных
приложений), без видимой ошибки — в логе `shopify app dev` была только строка
`Authenticating admin request`, без продолжения и без стектрейса.

**Причина**: только что добавленный `billing.check()` в loader'е
`app/routes/app.tsx`. План Shopify App Pricing ($9/мес) ещё **не создан** в
Partner Dashboard (это отдельный ручной шаг — см. раздел "Шаг 4" выше).
Без существующего плана `billing.check()` бросает необработанную ошибку;
необработанная ошибка в loader'е embedded-приложения приводит к тому, что
Shopify admin трактует приложение как неработающее и откатывается на список
установленных приложений — без явного сообщения об ошибке на экране.

**Промежуточное решение (было)**: `billing.check()` и связанный с ним редирект
на pricing_plans временно закомментированы в `app/routes/app.tsx`.
Подтверждено: после отключения `/app` открывается нормально.

**Финальное решение (текущее состояние кода)**: `billing.check()` возвращён и
обёрнут в `try/catch` с политикой **fail-open** — если проверка биллинга
бросает ошибку (пока нет плана в Partner Dashboard), доступ к `/app`
разрешается, ошибка только логируется через `console.error`. Редирект на
pricing_plans срабатывает только если `billing.check()` успешно отработал и
явно вернул `hasActivePayment: false`.

Это осознанный компромисс: строгий gate (падать/блокировать при ошибке
`billing.check()`) недопустим сейчас, потому что редиректил бы на
несуществующую страницу плана и ронял бы всё приложение — именно это и
произошло в описанном выше симптоме. Fail-open даёт корректное поведение в
обоих состояниях:
- **план не создан** (сейчас) — ошибка, доступ разрешён, приложение работает;
- **план создан, но нет подписки** — `billing.check()` успешен,
  `hasActivePayment: false`, происходит редирект на pricing_plans как и
  задумано.

**Оставшийся ручной шаг**: создать план Shopify App Pricing ($9/мес,
7-дневный trial, без капов) в форме сабмита приложения на
partners.shopify.com. Код меняться не должен — gate заработает штатно сам
по себе, как только план появится.

### Уточнение, 2026-07-20: `billing.check()` бросает `Response`, не `Error`

При живом прогоне (`shopify app dev`) пойманное в `catch` значение оказалось не
обычной ошибкой, а объектом `Response` со статусом **200** и телом HTML
(с CSP-заголовками `frame-ancestors` под iframe embedded-приложения) — то есть
похоже на служебную страницу SDK, а не на код ошибки в привычном смысле.

Гипотеза "это штатный `throw Response`-редирект, который нужно пробрасывать
дальше (`throw error`), а не глотать" была проверена и **отклонена на
практике**: после `throw error` вместо `console.error` страница снова начала
редиректить на список приложений — то же самое некорректное поведение, что и
до всякой обработки ошибок. Значит текущий swallow-подход (полностью глотать
и логировать, без re-throw) — не временный костыль, а единственно рабочее
поведение из проверенных на данный момент.

**Открытый вопрос** (не выяснено, пока план не создан в Partner Dashboard):
почему `billing.check()` вообще бросает `Response`, а не просто резолвит
`hasActivePayment: false` или бросает обычную ошибку. Возможно, это связано с
отсутствием сессии/токена в конкретный момент, а не с отсутствием плана как
таковым — стоит перепроверить это место после того, как план появится: если
ошибка исчезнет сама, значит дело было в конфигурации биллинга; если
`Response`-throw повторится и тогда, потребуется более глубокое исследование
(возможно, отдельный вопрос в Shopify community/поддержку).

## Исправлено, 2026-07-20: iframe-превью не грузился

**Симптом**: на странице приложения в секции превью — "Сайт
simple-dev-2s9mhugw.myshopify.com не позволяет установить соединение."

**Причина**: архитектурная ошибка на шаге 4 — витрины Shopify по умолчанию
отдают заголовки (`X-Frame-Options`/CSP `frame-ancestors`), запрещающие
встраивание в `<iframe>` с чужого origin. Admin UI приложения рендерится в
своём iframe внутри `admin.shopify.com`, что для самой витрины — чужой домен;
это защита от clickjacking на уровне платформы, не зависит от настроек
конкретного магазина и не обходится кодом приложения. Идея "живой iframe" из
шага 4 была нежизнеспособна с самого начала.

**Исправление**: `<iframe>` заменён на `<s-link target="_blank">`, открывающую
магазин в новой вкладке. Полноценного inline-превью триггеров в текущем виде
нет — компромисс, принятый вместо решения этой задачи (например, скриншот
через отдельный сервис) в рамках MVP.

## Диагностика, 2026-07-20: триггеры не срабатывают на витрине

**Симптом**: все три триггера включены в Admin UI, товар добавлен в корзину,
переход на другую вкладку — ни звука, ни изменения заголовка, ни попапа.

**Причина**: app embed блок "Micro-triggers" (шаг 3) ещё не был включён на
самом dev store через редактор темы. Это ручное действие, которое нельзя
выполнить через CLI/код — Theme App Extension по архитектуре Shopify требует
явного согласия мерчанта на встраивание в тему через UI редактора. Без этого
шага `assets/micro-triggers.js` физически не подключается ни к одной странице
витрины, поэтому все обработчики событий (`visibilitychange`, `mouseleave`,
`submit`) просто отсутствуют в DOM.

**Что нужно сделать вручную** (см. также раздел "Шаг 4" выше про то же
ограничение для dev store):
1. `npx shopify app dev` (если ещё не запущен) — подключит расширение к
   dev store.
2. В Shopify admin: **Online Store → Themes → Customize** (редактор темы) на
   `simple-dev-2s9mhugw.myshopify.com`.
3. В левой панели редактора — вкладка **App embeds** (иконка пазла обычно
   внизу списка блоков).
4. Найти "Micro-triggers" и включить тумблер.
5. Сохранить (**Save** в редакторе темы).
6. Добавить любой товар в корзину на витрине и только после этого проверять
   триггеры (переключение вкладки, движение курсора к верхней границе,
   добавление в корзину).

Пока этот шаг не сделан, отсутствие эффекта от триггеров — ожидаемое
поведение, не баг в коде.

## Добавлено, 2026-07-20: подсказка про app embed прямо на странице приложения

По итогам предыдущей диагностики стало ясно, что инструкцию про включение app
embed блока нужно показывать не только в docs, но и мерчанту прямо в Admin UI
— иначе неактивные триггеры выглядят как баг.

`app/routes/app._index.tsx`: добавлена секция "Enable on your theme" со
ссылкой-**deep link** в редактор темы, которая сразу открывает вкладку
"App embeds" с предвыбранным блоком "Micro-triggers" (мерчанту остаётся только
включить тумблер и нажать Save, не искать блок вручную):

```
https://{shop}/admin/themes/current/editor?context=apps&template=index&activateAppId={api_key}/app_embed
```

Формат подтверждён документацией Shopify (Theme App Extensions →
"App embed block deep linking"): `{api_key}` — `client_id` из

## Шаг 5 (QA) — начато, 2026-07-21: попытка настроить Billing план

**Что сделано в Partner Dashboard** (partners.shopify.com, не dev.shopify.com
— это два разных интерфейса; для этого проекта App Store listing и billing
остаются именно в старом Partner Dashboard, а построение/логи/версии
переехали в новый Dev Dashboard):

1. Приложение "simple-app" зарегистрировано в Shopify App Store
   (**App distribution → Register**), выбран режим **Public distribution**.
   Это необратимое (без обращения в поддержку Shopify) решение — подтверждено
   явно перед кликом.
2. Оплачен одноразовый регистрационный сбор $19 (Individual, привязана карта).
3. Дошли до страницы **App Store review → Preliminary steps** — чеклист
   технических требований перед тем, как откроется полная форма листинга
   (где, судя по документации, и находится Pricing).

**Почему Pricing/сабмит отложены**: чеклист "Preliminary steps" требует
закрыть сразу несколько вещей, которых у проекта пока нет:
- Реальный продакшн-домен/хостинг (`application_url` сейчас
  `https://example.com` — Shopify явно запрещает домены со словами
  "shopify"/"example" для сабмита; нужен реальный деплой, например
  Railway/Vercel + Supabase/Railway Postgres).
- App icon.
- Emergency contact (email + телефон).
- Primary listing language.
- Явное указание "My app won't use customer data" (актуально — приложение
  не читает customer data).

Это по объёму отдельная задача (деплой в продакшн + подготовка листинга),
не входящая в рамки "создать billing-план". Решено отложить прохождение
чеклиста и Pricing до момента, когда код будет полностью готов и появится
план хостинга — **billing.check() остаётся в fail-open режиме**
(см. раздел выше), доступ к `/app` не блокируется.

**Что делать, когда вернёмся к этому**: продолжить с той же страницы
(App Store review → Preliminary steps) в Partner Dashboard, начиная с
"Fix requirement issues with your configuration" → заменить
`application_url`/`redirect_urls` в `shopify.app.toml` на реальный домен.
